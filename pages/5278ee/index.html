如何准备技术答辩PPT？

# 当时遇到的挑战

答辩前，我把PPT给组里的资深同事A review，他告诉我这样的PPT去答辩不可能通过，因为他看不懂。而且在我们组这次去参加答辩的几个同事里（同级别），我的PPT写得最差。

听到这样的评价后有点慌，同事A的话非常直接和难听，虽然他没有提出具体的修改意见，但是一句“我看不懂”，已经直指要害了。

# 当时的思考、操作过程和结果

我认真考虑了他的观点，在没有任何人指导的情况下进行了较大修改。相比委婉好听但没有意义的评价，我更加尊重和感谢这样难听但有用的意见。

另外我也把PPT给了其他同事review，其中同事B提了一个大意见，说我里面的一个算法有点多余，是不是可以删掉？我当时一听就有点愣了，这个算法可是整个PPT里最重要最核心的内容啊，删掉了我拿什么去讲？为什么他会觉得多余呢？为什么这么重要的点，同事看不明白？于是我又详细跟他说了下原委，对算法部分进行了补充和增强。

这次答辩的过程非常顺利，我明显感觉答辩现场的评委被我吸引住了。甚至有评委当场提问“这个算法是你一个人独自提出来的吗？还是有其他同事提出，你负责实现的？”，当我说完“是的，这个是我独立发明并且实现的”，我就觉得这次肯定过了。

# 经验总结/感悟

答辩前准备

# 提前两周准备PPT

一般来讲答辩的时候都会准备一份PPT，当然如果你非常相信自己的表达和技术能力，不准备PPT，现场裸讲也行。但是没有PPT的辅助，单靠口语描述，很难将一些复杂的结构和流程表达清楚，所以强烈建议大家用心准备一份PPT。

到底该提前多长时间准备PPT呢？这个没有标准。我有见过上午准备PPT，下午就答辩并且通过的。也有见过准备了数周甚至数月PPT的，最后没有通过的。

所以还是根据个人情况决定吧，通常提前至少两周以上比较保险。一周时间写PPT，一周时间调整和演练。如果不是特别有信心，或者完全没有经验，那就再多花点时间。

我在准备PPT时很少占用过工作时间，基本是上下班路上，回家晚上，或者周末。

# 一个人的演练

PPT写完了需要演练，如何进行呢？

找一个安静无人的地方，比如晚上的会议室或者下班之后在家里，powerpoint的放映方式选择“排练计时”，这样每次演讲完后都知道每页PPT的耗时分布，方便自己调整。 一定要说出来，不要在心里默念。 注意页和页之间的衔接，一定要非常清楚下一页的内容，一定要有逻辑地进入下一页。 不要每次都从第一页开始演练。 上下班路上可以利用空闲时间在心里默念PPT的内容，非常有助于脱稿演讲。

# 你不是一个人在战斗

PPT写完后，一定要请资深的同事或者leader review你的PPT。特别是当你还没有很多经验时。 一个人练熟了，可以在leader或者资深同事们面前公开讲一次。请他们指出你的问题，比如语速，表情，手势等。

公开试讲还是挺重要的，特别是较少在公开场合演讲的同学，一定要加强这方面的锻炼。否则见到评委，还没开始就紧张了，很容易影响自己的发挥，输在怯场上就太吃亏了。

# 脱稿而出，心中自有PPT

一直讲，一直练，直到你不看PPT也能把内容非常连贯地讲出来为止，做到真正的脱稿而出。为什么呢？因为答辩现场的局面你完全无法控制，有很多突发情况会打断你的思路和表述，如果不熟练，很容易慌乱，甚至导致你大脑空白思维混乱，吞吞吐吐词不达意。

在你讲的兴起的时候针对你的技术方案提出尖锐问题，这是最正常和友好的交互。 评委有可能提出明显不信任的质询，“这个是你负责的吗？这个算法是你独立提出的吗？这个方案是业界第一个吗？”。 也有可能提出改进性建议，“不要讲这么快，我们思路跟不上”，然后我转头看几个评委，发现他们也是一脸苦闷懵逼，于是降低语速，却又担心时间不够。 还有可能投影仪连接异常，PPT屏幕突然黑了，怎么保持正常的心态？ 稍微不友好地，评委会不耐烦直接打断你，“这个太简单了，说重点”，“不要说些虚的东西，说说你的实际工作”，你慌不慌？我第一次答辩的时候就被评委这样叫停过。

如果你不需要PPT也能将内容非常连贯地陈述出来，心中自有PPT的话，那上述突发情况你都能很快应付，即使被突然中断，就算评委的问题回答不上，那也能很快的收拾混乱的思路，将你的项目主线和演讲逻辑，迅速地衔接起来。

不要问我怎么知道的，因为很多情况我都遇到过。每次被打断，心里都会有很多念头，“糟了，是不是没讲好？我是不是要挂了？怎么会这么问，好像不相信我说的话？”，后来准备充分了，这样的念头都是一闪而过，几秒钟就能恢复正常。

还有很重要的一点，答辩前将PPT充分演练，做到脱稿演讲，能够极大地增加你的自信，也不会特别怯场了。

# 如何撰写PPT

上一节说要花至少两周以上的时间准备PPT，那PPT该写些什么内容呢？什么该多写？什么该少写甚至不写呢？

“重点突出，逻辑缜密，条理清晰，层次分明，结构严谨，用数据说话，结果第一，图表丰富等”。道理大家都懂，但依然写不好PPT，答不好辩。

接下来我就用具体，形象的文字来表达一下我的观点。

# PPT的版式—-简洁统一

技术答辩的PPT不需要花哨的外表和形式：

不需要动画。不需要各种飞舞和横入。 不需要华丽的表格图片。 不需要浮夸的字体。

为什么不需要呢？

关注重点不同。技术通道的评委更加关注的是你的技术亮点。比如架构，算法和性能，而不是PPT的外表。 浪费时间。花费很多时间去设计精美的PPT，浪费自己的时间，有这个精力还不如多审视一下自己的内容。 不方便演讲现场的页面控制。如果评委需要回退到某页PPT，加了动画反而会很麻烦。 不好把控美的力度。大部分技术同学缺乏深厚的PPT功力，审美观也很一般，你所谓的美和华丽，在内行看来可能只是“不会穿衣服的花姑娘”。 评委不一定能欣赏。就算真的漂亮，由于大家都是技术出身，评委也有可能看不懂，欣赏不了你PPT的美。

当然我只是说不需要，不建议，没有说不允许。相反，如果你对自己的PPT设计能力和审美都非常自信，那尽情玩吧，在技术方案同等出色的情况下，这样的PPT说不定能给你加分呢。

  • 大部分技术通道的同学在PPT的形式上只需要注意“简洁统一”即可：

字体和配色统一。这里说的统一，都是针对同一级别的。千万不要出现同一级别的标题或者正文出现大小不一的字体，同一段内容或者页面出现很多颜色。 注意排版的齐整。同一版面的内容不要出现错乱的对齐方式，居中就严格居中，不要弄得左对齐也不像，居中也不是。 表格，流程，架构注意添加标题，数据最好注明来源。 需要特别强调的地方，可以用粗体，高亮醒目的颜色标示一下。 还有一些基本的技巧，比如不宜出现大段描述文字，架构和流程需要图形化表示等，不再赘述。

其实写好PPT很不容易，里面的学问很大，我没有资格教大家写好PPT。简洁统一，肯定不是最好的表述方式，但一定是非常经济实用的方式，甚至是性价比最高的方式。它不仅仅能够节省你浪费在PPT上的时间，

而且会让评委聚焦在你的内容上，显得更加专业，干货也更加突出。

对于我来讲，使用统一的模板，白底黑字，已然足够。

# PPT的内容—-系统、全面、深入

选择了简洁统一的PPT版式，重点来了，如何编写项目内容呢？

  • 内容的取舍

由于答辩时间通常为30分钟左右，还包括评委提问时间，所以PPT的内容肯定是有限制的。于是就会经常遇到这两种情况：

内容太多了，时间不够讲不完。 内容太少了，时间剩余又显得太单薄。

  • 内容太多了怎么办

这个问题我以前也很犯难，做得事情太多了，都想讲，经常超时讲不完。不过我记得有一次，一个同事看了我的PPT之后，给我提了一个问题，“你的内容太多了，如果只允许你讲一个或者两个项目，你会怎么选择？”。我恍然大悟。

后来再进行内容取舍时，我就一直这样问自己，“如果只能讲两个项目，会选择哪两个呢？”，逼自己做出选择。

很多时候，很多项目之间的差距并不明显，那具体该选择哪一些呢？一般来说，需要围绕两点做出权衡和选择，“技术深度和收益”。我简单说下自己的判断：

技术深度。有没有深度，自己一般会有个初步判断，哪一个花的时间多，做起来吃力，基本上就是更加有深度的了。但不要局限在自己的认识里，因为你觉得难的，可能在别人眼里觉得很简单，自己一定要做横向对比。实在拿不定主意，请leader帮忙。 技术收益。有时候几个项目之间的深度和难度好像没什么差别，那就优先讲有产出的，特别是跟KPI相关的。范围由大到小，部门->团队->个人。

  • 内容太少了怎么办

我还没有遇到内容太少的情况。如果你遇到了，很有可能是你平时不注意项目和数据的积累。建议：

重要的项目，一定要写设计文档。 一些独到的经验可以在KM上分享。 重要的邮件和数据记得备份。 可以主动在部门，公司甚至业界进行技术分享。 到了快答辩的时候，把上面的内容整合一下，就有了答辩材料。

# 具体要求

  • 主题明确

每一页都应该有一个，而且只有一个，非常明确的主题。可以制作成标题，让评委一眼就知道你这页PPT在讲什么内容。

如果内容太多，那就继续拆分，制作成多页PPT和多个主题来描述。

  • 系统

什么叫系统呢？简单说它必须是一个有逻辑的完整故事。

一定要有原因和背景，你为什么做这个项目，它的意义是什么。这里面需要特别注意，有些背景需要重点介绍，有些背景需要一带而过。 一定要有过程，如何实现的，架构，流程等。你具体做了哪些工作。 一定要有结果，有哪些收益和产出？对于一些正在进行的项目，可能还没有收益。但肯定有阶段性成果和产出。比如设计方案、架构、模块等。

  • 全面

全面的意思很容易理解。但是在技术答辩的时候，面面俱到是没有必要的，一些小的细节更加没有必要描述，时间上也不允许。这里的全面，我想强调的是两点：

  • 知己知彼。
  • 风险异常。

# 知己知彼

什么叫知己知彼呢？就是要有调研对比。要知道别人的方案是如何实现的。比如部门内，公司内，业界的通常做法。你的方案有什么特点和优势。

这个是非常重要的，上去讲一个项目，结果评委一提醒才发现别人早就做好了，或者业界早就有了成熟的解决方案。你为什么还要重复造轮子？背后有什么特殊的考虑吗？方案完成后，收益和数据相比其他方案的区别在哪？

如果这一点都没有调研清楚，已经没有必要继续往下说了，很大概率直接挂掉。

  • 风险异常

任何一个技术方案，都可能出现故障，发生风险。所以架构设计上，要重点考虑这方面的问题。面试答辩，评委问起各种异常的时候，要做到心里有数。比如机器死机了，流量突涨了，网络故障了，系统如何提供正常服务？这都是很常见的问题。

  • 深入

深入简单点说就是要有深度，有难度。不仅仅包括工程架构方面的技术难度和深度，其他的一些工作，比如精巧的细节把握，数据的运营分析等，都可以反映出项目是否做得深入。

  • 技术难点

  • 细节把握

  • 数据运营

  • 技术难点

技术难点很好理解，比如复杂的数据结构，高效的算法改进，强悍的吞吐性能，分布式系统的强一致性和实时性保证等。但遗憾的是，不是所有项目都能一眼看透技术难度。比如新方案的预研，偏业务和运维的项目，那这些项目如何深入描述呢？

  • 精巧细节的把握

项目的主线有了，但有些细节往往也能体现技术能力，我举例说两个小的细节：

1、大家应该都听说过SPDY和HTTP2了，它们的一个重要特性就是头部压缩。那头部压缩有什么作用呢？这个问题好像很简单，压缩嘛，数据传输量通过压缩变少了，传输肯定就快了，原理上这么说完全正确。但真实的应用场景和遇到的问题是这样的：

头部压缩最重要的是能够明显提升上行数据的传输效率。为什么？因为上下行带宽严重不对称，上行带宽往往只有下行带宽的十分之一甚至几十分之一。 而WEB页面的平均头部大小有将近1500个字节，所以下行带宽有100K的话，上行带宽往往只有1K或者几百个字节，所以带宽层面就限制了上行数据的传输效率。如果有90%的头部压缩比就能显著提升上行数据的传输效率，而下行数据呢？由于下行带宽一般足够，所以头部压缩的提升效果有限。 如果进一步思考的话，为什么上下行带宽会严重不对称呢？这又是个比较复杂的问题，一方面是传输信道的规定（国际电信组织制定的标准），另外一方面是运营商出于节省成本的考虑，可能会做出更加离谱的带宽比。 那如何利用头部压缩的特性呢？ 由于SPDY使用的zlib的deflate算法，这个算法的特点就是信息越冗余，越重复，压缩比越高。通过实验无意中观察发现，第一次传输的头部压缩比只有30%，第二次传输的压缩能达到60%，只有第三次及之后的压缩比才能达到90%。 那如何提升用户的压缩比呢？ 可以使用预连接。即用户打开页面之后，首先发送两个空的请求，积累重复的头部数据。当用户真正发起请求时，已经是第三个请求了，压缩比一下就能达到90%。

2、熟悉HTTPS协议的同学都知道SSL3.0协议已经不安全了。但是部分老的客户端，比如IE6，只支持SSL3.0。由于业务的要求，我们还必须支持SSL3.0。如何支持SSL3.0呢？

首先就要分析SSL3.0为什么不安全，主要是两点： 1）aes-cbc存在POODLE漏洞。 2）RC4存在FREAK漏洞。 那哪个漏洞危害更严重呢？POODLE。所以我们针对强需求业务开启SSL3.0，但是只支持RC4，关闭AES算法。

上述两点，HTTP2头部压缩的特性，SSL3.0不安全，只要稍微接触相关知识的同学都非常清楚，可谓广为人知。但是背后的细节，又有多少同学掌握得很清楚呢？

表面上看这些工作的量很小，甚至只是一个配置开关的启用，既没有设计方案，也没有开发代码，更加没有看起来高大上的架构流程说明。但是第1个细节，可以看出实验数据的观察和分析能力，压缩算法，前端页面及用户行为的理解。第2个细节，也是需要对协议的安全性有较深的认识。

这2个小的细节，完全能够体现严谨认真的工作态度和扎实深入的技术功底。也正是这些小细节，体现了完全不一样的工匠精神和技术层次。

  • 数据的分析运营

有很多工作是跟数据有关，特别是运营系统和业务系统。

之前在做用户访问速度优化的时候遇到过一个问题，就是客户端的操作系统，浏览器种类非常多，再加上服务端协议版本，密码套件等特征，影响访问速度的因素非常多。那如何提升用户的访问速度呢？通过如下三个步骤：

数据采集。通过JS和客户端采集几百个维度的速度数据，比如 “X5内核在4G网络下使用HTTP2并且是TCP第一次连接并且使用了ECDHE密钥交换算法并且没session resume的访问速度”是多少？ 数据分析。分析不同维度，相同维度下的各个速度数据，找出瓶颈。 数据运营。采集和分析了这么多的数据，有什么作用呢？如何运营呢？同样有几个点： 1，推动服务端配置和性能优化。哪些算法慢，哪些耗费资源，就不要使用。 2，推动业务侧优化（包括前端页面和App侧）。域名收敛，预连接建立等。 3，探索速度优化的方向。QUIC，对称加密优化，异步代理等。

同样的，这里的数据运营和分析，并没有什么高深复杂的技术，几个简单的脚本就能够完成全部工作。但这个工作完全能够体现出综合的技术能力，TCP协议，HTTP协议，SSL协议等，不掌握这些基础就不可能设计出这些数据维度，也很难优化访问速度。谁说数据分析和运营没有技术含量？相当有。

  • 禁忌点

一些非常明显的禁忌我就不做介绍了，比如造假，剽窃。接下来说一下PPT中不经意间容易出现的大忌。

多 ≠ 好

项目内容不是越多越好。重点介绍2个项目，最多3个，足够了。如果工作量确实很多，就把它们放在一页PPT里整体介绍下就可以。

通过一两个重点项目亮出你的技术高度，再通过其他项目的罗列辅助证明你充实的工作量。每个都当做重点讲，很有可能一个都讲不好，因为时间只有那么多，每个项目都只说个大概，没有亮点和深度，适得其反。

技术答辩中的重点项目就类似考试中的附加题一样，做出来了，足以证明你的能力不错。所以项目不在于多，在于深。

  • 流水账

上去就讲自己做了什么项目，设计了什么架构，实现了哪些功能，应用在了哪些场景，产出了哪些收益。没有方案调研，没有深入分析，没有难点阐述，没有风险描述，没有收益对比等。然后继续讲下一个项目，同样的套路，平淡无奇，中规中矩，好像没什么毛病，但总感觉少了些什么。

就像小学生日记一样，今天天气很晴朗，去了植物园参观，看到花很漂亮，然后回家吃饭，写日记。这是一篇优秀的日记吗？显然不是。

同样的，流水账式的技术答辩会让评委觉得很平淡，毫无亮点和新意。这样的陈述，很难通过。评委想看到的不是你做了哪些事情，而是你做了哪些漂亮的事情。

  • 缺少推导

我在上次答辩中也犯了这个错误。评委说很多结论没有推导过程。事实上这是我一直都牢记在心的禁忌，但由于时间太少，语速太快，很多推导过程完全来不及介绍，给评委留下了不太好的印象。

为什么这个是大忌呢？

台上一秒钟，台下十年功。同样地，结论一句话，但背后付出的思考调研，设计分析等，要耗费非常多的时间精力，也是能体现一个人解决问题，创新思维的关键。

结论就好比海上冰山显露出来的一角，而背后的推导就如同海底冰山的全貌，你不展示出来，谁知道你的分量有多重呢？

再说了，直接给结论，万一你是抄袭过来的或者照搬别人的呢？不说推导过程，如何证明这个是你的工作呢？
